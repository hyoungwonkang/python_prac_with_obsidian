# detection-ai-study — 탐지 플랫폼 대응 학습 트랙

## Context

- 업무 프로젝트: **아동·청소년 온라인 성착취 의심정황 탐지 AI 플랫폼** (Hybrid RAG + Graph RAG 중심 고도화 아키텍처, 2026-07-02 공유).
- 파이프라인: 수집 → 원본 보존 → 추출(OCR·임베딩) → **1차 탐지(Rule·KoBERT 분류·NER·이미지)** → 근거 검색(Hybrid RAG) → 관계 분석(Graph RAG) → LLM 판단·스코어링 → 검수 → 재학습(MLflow).
- [[llm-from-scratch]] (교재 트랙)가 마무리 단계에 들어서며, **플랫폼의 1차 탐지 계층을 코드로 재현할 수 있는 것**을 다음 학습 목표로 설정.
- **학습 순서의 배경**: 이 우선순위를 정해준 상사가 BERT 학습 전에 교재(Raschka)를 먼저 권함 — 트랜스포머 내부·파인튜닝 메커니즘을 바닥부터 다진 뒤 BERT 계열에 진입하는 의도. 우선순위와 교재 선행 모두 같은 상사의 로드맵.
- 아키텍처 원본 이미지는 업무 자료라 **vault에 커밋하지 않음** — 로컬 보관 위치는 `30-References/_업무자료-로컬/아동청소년-탐지플랫폼-고도화아키텍처.png` (gitignore 등록, Obsidian 로컬 열람용. 다른 PC에서는 업무 채널에서 다시 받아 같은 경로에 배치).

#### 지시 4′ 해석 — 성공과 실패가 각각 다른 것을 증명

두 결과를 합치면 이 실험이 증명한 것은 **① "직접 라벨링→학습→탐지 루프가 실제로 돈다"(성공)** 와 **② "커스텀 클래스를 범용 모델에 억지로 합치면 기존 능력이 무너진다"(실패의 교훈)**. 성공과 실패가 서로 다른 것을 증명한 것이 핵심.

**결과 1 — mAP50 0.995의 진짜 의미와 함정**

- **왜 16장으로 0.995인가**: COCO 사전학습 백본이 이미 선·질감·형태를 알고 있어(전이학습), 모델이 새로 배울 것은 "잠옷이 어떻게 생겼나" 하나뿐. 그래서 소수 샘플로도 빠르게 습득.
- **함정(도메인 협소성)**: 검증셋이 학습셋과 **같은 분포**(같은 방·조명·사람·웹캠)에서 나왔으므로, 0.995는 "학습 때와 거의 똑같은 조건에서 찾을 수 있나?"를 측정한 것이지 "처음 보는 환경에서도 찾나?"가 아님. 즉 **파이프라인이 end-to-end로 작동한다는 증명**이지 성능 완성이 아니다. 지시 4′의 목적(라벨링→등록→학습 루프 실증)은 정확히 달성.

**결과 2 — COCO 붕괴 0.1768: 원인은 사실 두 개가 겹침**

- **① 파국적 망각(catastrophic forgetting)** — 일반 현상. 신경망엔 기억 보호 장치가 없어, 새 데이터로 파인튜닝하면 경사하강법이 잠옷에 유리한 방향으로 가중치를 옮기는데 그 가중치가 COCO 클래스를 담당하던 것과 **공유**되어 옛 지식이 덮어써짐.
- **② 라벨 결손 노이즈(label-deficit)** — 이 실험의 진짜 주범. 직접 찍은 잠옷 16장엔 **COCO 물체가 라벨링돼 있지 않다**(프레임에 의자·사람이 있어도 라벨 없음). 그런데 객체 탐지에서 **라벨 없는 물체 = 암묵적 배경(negative)** 이므로, 모델은 "실제로는 의자인 영역을 배경으로 간주하라"고 **능동적으로 잘못 배운다**. 즉 16장 잠옷 데이터가 "의자·사람 자리는 배경"이라며 COCO 탐지 능력을 억누르는 독이 됨.
- 그래서 0.1768은 단순 "잊어버림"이 아니라 **"잊어버림 + 잘못된 교육"이 겹친 값** → 망각 예상치보다도 더 나쁘게 떨어진 이유.

**왜 결론이 "병행 운용(YOLO_PT 교체)"인가 — 대안 지도**

| 방법 | 실효성 | 이 실험에서의 판단 |
|---|---|---|
| 전체 재라벨링(모든 이미지에 모든 클래스 라벨) | 근본 해결 | 손라벨 16장에 COCO 80클래스까지 다 달기는 비현실 |
| 리플레이/리허설(원본 COCO 일부를 섞어 재학습) | 망각 완화 | 유효하나 데이터·시간 비용 발생 |
| 백본 동결 + 헤드만 학습 / LR 낮추기 | 부분 완화 | 라벨 결손 노이즈 자체는 못 막음 |
| **모델 분리·스위칭(병행 운용)** ← 채택 | 실무 정답 | 니치 커스텀 클래스 하나 붙이는 데 가장 싸고 안전 |

**플랫폼 관점 연결**: 이 실험은 지시 대전제인 **"모든 모듈은 도메인 교체 가능하게 설계"**([[#새 지시 목록 2026-07-08 — 기존 로드맵과의 매핑|지시 목적]])와 맞물린다. "새 도메인 = 기존 모델에 병합"이라는 본능이 왜 틀렸는지를 데이터로 보여줬고, 대신 **모듈/가중치 분리 + 스위칭**이 옳은 방향임을 실증 → 실패 실험이 아키텍처 설계 원칙(모듈 독립성)을 뒷받침하는 근거가 됨.

### 후순위 (플랫폼 2단계 도입 시점에)

- **Hybrid RAG** (SBERT 문장 임베딩 → 벡터 검색 → BM25 하이브리드 → Reranker) — 아키텍처 §2 MVP 1단계 핵심(④ 근거검색). 위 목록 완료 후 진입.
- Graph RAG (Neo4j·Entity Linking, ⑤) · **정형 판단 LLM/SLM(⑥, 이전 phi-2 자리 — 특정 모델 미정)** · MLOps 심화(Model Registry·Canary) · 인프라(K8s·Kafka·OpenSearch)
