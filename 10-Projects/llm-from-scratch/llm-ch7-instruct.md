## 학습 메모 (실습으로 익힌 것)

### 데이터 준비 (`instruction_dataset_finetune.py`)
- 다운로드: rasbt 레포 `instruction-data.json`(1100건). `import urllib.request` 필요 — `import urllib`만 하면 `urllib.request` 미로드(AttributeError). 실제 겪음.
- `format_input`: Alpaca식 프롬프트 템플릿("Below is an instruction..." + `### Instruction` + 선택적 `### Input`). 응답은 `### Response`로 별도 결합.
- 분할: 단순 **순서 슬라이싱** train 85%/test 10%/val 나머지. ch6은 `random_split`(섞음) — ch7은 순차 분할(데이터가 이미 다양). val은 `[...:]`로 나머지 전부(int 버림 누락 방지).

### 콜레이트 함수 = 배치 시점 처리 (단계 2.x) — ch6 DataLoader로 부족
- 데이터 준비(다운로드·format·분할)는 **콜레이트 전 단계**. `InstructionDataset`(토큰화)이 단계 1, **콜레이트 함수**가 단계 2.x.
- ch6은 입력만 패딩하면 끝(라벨=숫자). ch7은 입력·타깃 둘 다 자유 텍스트라 콜레이트에서 ①배치 내 최장 길이 패딩 ②**타깃=입력 한 토큰 이동** ③패딩 자리 **-100**(손실 제외)까지 직접 처리 → 그래서 사용자 정의 콜레이트.

### 타깃 = 입력 한 토큰 이동 = 다음 토큰 예측 (사전훈련과 동일)
- `입력[A B C D]` → `타깃[B C D E]`(한 칸 밀림). 각 위치에서 "바로 다음 토큰" 맞히기 = ch2 슬라이딩 윈도·ch5 사전훈련과 **같은 메커니즘**.
- "한 토큰 이동"(학습 방식) ≠ "GPT가 토큰 하나씩 다룸"(causal/자기회귀 구조)이지만 **원인-결과**: 이렇게 학습해서 → 자기회귀 생성이 가능.

### 분류 FT(6장) vs 명령어 FT(7장)
| | 분류(ch6) | 명령어(ch7) |
|---|---|---|
| 출력 | 고정 클래스(0/1) | 자유 텍스트 생성 |
| head | 768→2 **교체** | 768→50257 **유지** |
| 쓰는 출력 | 마지막 토큰 argmax | 토큰 하나씩(자기회귀) |
| 손실 | 마지막 토큰 CE | 모든 위치 다음 토큰 CE |
| 비유 | 객관식 | 서술형 |

### 로짓 = "어휘사전 각 칸의 점수 벡터" (2026-07-02 Q&A로 체득)
- **어휘사전 = 번호 붙은 토큰 명단**(0~50256, 고정·숫자 없음). **로짓 = 모델이 그 명단 전원에게 매긴 "다음에 올 것 같은 정도" 점수**. 명단은 로짓의 **틀**(칸 수·칸의 의미)만 제공, **값은 모델 계산**(`model(input_batch)`)이 만듦 — heads.py:123.
  - 비유: 어휘사전 = 선거 후보 명부(번호만), 로짓 = 득표수(투표 = 모델 계산, 매번 다름).
- 로짓 표의 두 축: **행 = 위치**("문장의 몇 번째 자리에서 다음은?"이라는 질문, 입력 길이만큼), **열 = 칸**(어휘사전 후보 번호, vocab_size로 고정). 숫자 하나 = "위치 △에서 후보 ○번이 올 점수".
  - `torch.tensor([[a,b],[c,d]])`에서 **안쪽 리스트 개수 = 위치 수, 리스트 속 숫자 개수 = 칸 수**. 토이 예제 = 위치 2 × 칸 2 미니 세계 (별도 '명단' 객체는 없음 — cross_entropy가 행 폭으로 후보 수 추론).
- 모델 내부에서 로짓이 태어나는 곳 = **마지막 출력 층**: 토큰 ID → 임베딩(768차원) → 트랜스포머 12블록 → **Linear 768→50257**이 위치마다 점수 벡터 생성.

### target = "정답 칸 번호표" + cross_entropy 채점 (loss 1.1269 분해)
- `targets = [0, 1]`은 **좌표가 아니라 답안지 두 줄**: "위치 1의 정답 = 0번 칸, 위치 2의 정답 = 1번 칸". target 길이 = 로짓 행 수, 값 = 정답 열 번호. (처음에 `[행0, 열1]` 좌표로 오독했음.)
- **예측 ≠ 정답**: 예측 = 로짓 최고점 칸(모델 생각), 정답 = 데이터가 정한 칸(target). 둘의 어긋남이 벌점의 근원. 토이 예제 위치 1이 일부러 "틀린 경우"(예측 칸1, 정답 칸0).
- cross_entropy 3단계: ①softmax로 확률화 ②**정답 칸의 확률만** 꺼냄 ③−log 벌점. loss **1.1269** = (틀린 위치 2.1269 + 맞힌 위치 0.1269) / 2. loss=0 = 정답에 확률 100%. **MLflow의 train_loss/val_loss가 정확히 이 계산.**
- **-100 = ignore_index**: target이 -100인 위치는 채점에서 완전 제외. 패딩 자리에 적용(패딩 맞히기 학습 방지). 행을 추가해도 -100이면 loss 불변(1.1269 유지)으로 확인.
- `allowed_special={"<|endoftext|>"}`: tiktoken은 특수 토큰 문자열을 기본 **거부**(데이터에 우연히 섞인 걸 진짜 신호로 오해 방지) → 의도적 사용임을 명시해야 ID 50256 하나로 인코딩. ch7 패딩 토큰으로 사용.

### 겪은 버그 3종 (콜레이트·Dataset 작성 중)
- `stack expects a non-empty TensorList` → **엉뚱한 리스트에 append**(`inputs_lst.append(targets)`). 교훈: 빈 리스트 stack 에러는 그 리스트에 append하는 곳을 역추적.
- `encoded_tests` 오타(→`encoded_texts`), `__getitem__`/`__len__`이 **`__init__` 안에 중첩**(들여쓰기) — 정의 시점엔 안 터지고 DataLoader가 부를 때 터지는 지연 폭탄류.

### 연습문제 7.2 — 지시 마스킹 (구현 요점)
- 목표: **채점 범위를 응답 구간으로 좁히기** — 패딩에 쓴 -100을 지시문+입력 구간에도 적용 (지시문 암송 학습 방지).
- 구현 3곳: ①Dataset이 `len(tokenizer.encode(instruction_plus_input))` 저장 ②`__getitem__`이 (길이, 토큰열) 반환 + 콜레이트가 쌍으로 언패킹 ③패딩 마스킹 뒤 `targets[:instruction_length-1] = -100`.
  - **-1인 이유**: target은 한 칸 밀린 것 → `targets[L-1]` = **응답 첫 토큰** 맞히기 → 남겨야 함.
- 단, 2026-07 현재 "마스킹이 무조건 정답" 아님 — 이 데이터(1,100건·짧은 응답)는 비마스킹 우세 조건. **두 방식을 MLflow run으로 비교가 올바른 과제 자세** → 상세: [[../../30-References/instruction-masking-trend]].

### 배치 처리가 빠른 이유 (GPU 병렬)
- 여러 샘플을 한 텐서로 묶어 넣으면 GPU **코어 수천 개**가 동시 계산 → 1개 처리 시간 ≈ 배치 처리 시간(노는 코어를 채움). 버스 빈 좌석 채우기 비유.
- **코어 ≠ 메모리**(독립 자원, 비례 안 함): 코어=계산 일꾼, 메모리=데이터 창고. 배치 키우면 노는 코어 채워 빨라지지만 **메모리 초과(OOM)** 면 멈춤 → 둘의 균형점에서 batch_size 결정.
