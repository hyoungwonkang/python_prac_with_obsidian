## 사례별 상세

### 사례 1 — TF mutex 크래시 재발 (시끄러운 실패)
- **증상**: `libc++abi: mutex lock failed` — 스크립트 본문 실행 전에 즉사.
- **원인**: 교재 코드를 그대로 치면서 `from gpt_download import ...` 부활 → 상단의 `import tensorflow`가 로컬 파이썬 3.9에서 C++ 레벨 크래시 (6장에서 이미 겪고 우회한 문제의 재발).
- **교훈**: 전역 설정으로 못 막는 import-시점 크래시 → **새 장 시작 때마다 gpt_download → 어댑터 치환이 표준 절차**.

### 사례 2 — 조용히 잘못 학습된 1.97분 (조용한 실패, 최악 유형)
- **증상**: 학습이 빠르게 "성공"했는데 생성이 "arcity Franch" 반복 횡설수설. 에러 없음.
- **진단**: MLflow 곡선 소급 확인 → **시작 loss 20** (정상 ~3.8, 무작위 10.8보다도 높음) = 시작부터 계산이 망가짐. 같은 배치로 CPU 3.81 vs MPS 19.77 → 장치 문제 확정 → forward 이분탐색 → 첫 블록 `W_query`에서 발산 → **비연속(non-contiguous) 가중치 + MPS 행렬곱 = 조용한 오답**.
- **원인**: `assign`이 numpy 전치 뷰를 그대로 텐서화 → stride 꼬인 Parameter. CPU는 정상 처리, MPS는 에러 없이 틀린 값.
- **조치**: `np.ascontiguousarray` 한 줄. 수정 후 CPU/MPS loss 일치(3.8112/3.8113).
- **교훈**: ① **시작 loss가 사전훈련 기대값인지 항상 확인** ② 값의 기대치를 알아야 조용한 버그를 잡는다 ③ MLflow 기록 덕에 소급 진단 가능 = 실험 추적의 실전 효용.

### 사례 3~6 — OOM 4연전 (한 꺼풀씩 벗겨진 원인)
- **공통 증상**: `RuntimeError: MPS backend out of memory`. 점진적 둔화 없이 한도 넘는 순간 즉사.
- **왜 교재 데이터는 무사했나**: 배치 폭 61~91토큰(모양 종류 적음·작음). Alpaca는 폭 40~512로 다양 + 긺.
- **원인의 계층** (시도가 실패할 때마다 다음 층이 드러남):
  1. 긴 샘플 → 배치가 큼 → **천장(512)으로 잘라도** …
  2. 피크 문제가 아니었음: **배치를 반감해도 같은 지점에서 죽음** = 스텝이 쌓일수록 자라는 **누적** 문제 →
  3. MPS 캐시 할당자는 **배치 모양(shape)마다 버퍼를 따로 캐시** → 동적 패딩이 만드는 다양한 모양만큼 증식 → `empty_cache()`로 청소했지만 …
  4. PyTorch 풀은 줄었는데(`MPS allocated` 28→19GB) **풀 밖 `other allocations`가 폭증**(15→28GB) = MPS가 모양마다 컴파일해 **영구 캐시하는 연산 그래프** — empty_cache로 못 비움.
- **최종 처방 (검증 대기)**: **버킷 패딩** — 배치 폭을 64의 배수로 올림(64~512, 최대 8종). 모양 종류를 원천 제한해 그래프 캐시 증식 차단. 단위 테스트 통과, 완주 검증은 미완.
- **교훈**: ① OOM 대응 = "동시에 올려두는 양 줄이기"의 변형들 (배치↓·천장↓·모델↓·캐시 차단) ② **배치 반감에도 같은 지점에서 죽으면 피크가 아니라 누적**을 의심 ③ 동적 패딩의 "같은 결과를 더 싸게"는 장치에 따라 대가가 다르다 — MPS에서는 모양 다양성이 역으로 메모리 비용 ④ exit code가 파이프(tail)에 가려질 수 있음 — 백그라운드 실행은 파이프 없이.

## 용어 정리

| 용어 | 뜻 |
|---|---|
| **OOM** (Out Of Memory) | 메모리 부족으로 강제 종료되는 에러. 점진적 둔화 없이 한도 넘는 순간 즉사 |
| **MPS** (Metal Performance Shaders) | 애플 실리콘 맥의 GPU 가속 백엔드. `device="mps"`. CUDA(NVIDIA)의 맥 대응물 |
| **통합 메모리** (unified memory) | 맥에서 CPU와 GPU가 공유하는 하나의 메모리(이 머신 36GB). GPU 전용 VRAM이 따로 없음 |
| **contiguous / 비연속** | 텐서 값이 메모리에 순서대로 나란히 있는가. 전치(transpose) 등은 값 복사 없이 **읽는 순서(stride)만 바꿔** 비연속이 됨. CPU는 비연속도 정상 계산, MPS 행렬곱은 조용히 오답 (torch 2.8.0 실측) |
| **stride** | 텐서에서 "다음 행/열로 가려면 메모리를 몇 칸 건너뛰나". 비연속 텐서 = stride가 자연 순서와 다른 텐서 |
| **캐시 할당자** (caching allocator) | PyTorch가 GPU 메모리를 매번 OS에 반납하지 않고 쟁여두는 장치. MPS에서는 배치 모양마다 따로 쌓음 |
| **`torch.mps.empty_cache()`** | 캐시 할당자가 쟁여둔 미사용 블록을 반납. **연산 그래프 캐시는 못 비움** |
| **연산 그래프 캐시** | MPS가 (연산 × 텐서 모양) 조합마다 컴파일해 영구 보관하는 실행 계획. 모양 종류가 많을수록 무한 증식 → 이번 OOM의 최종 원인 |
| **동적 패딩** | 배치 폭을 "그 배치의 최장"에 맞춤. 계산 낭비 0, 배치 모양 무한 |
| **버킷 패딩** | 배치 폭을 정해진 배수(64)로 올림. 낭비 10~20%, 모양 종류 유한(8종) — 동적↔고정의 절충 |
| **천장** (`allowed_max_length`) | 시퀀스를 이 길이에서 잘라내는 상한. 패딩의 "바닥 맞추기"와 별개인 "천장 자르기" |
| **활성값** (activations) | forward 중간 계산 결과. 역전파에 필요해 쌓아두며, **배치 크기 × 시퀀스 길이에 비례** — 학습 메모리의 가변 부분 |
| **옵티마이저 상태** | AdamW가 파라미터마다 유지하는 보조값 2개. 355M이면 가중치(1.4GB)의 2배(2.8GB)를 상시 점유 |
| **mutex 크래시** | 이 로컬 파이썬 3.9에서 `import tensorflow` 시 나는 C++ 크래시 (`mutex lock failed`). OOM과 무관한 별개 문제 |
| **조용한 실패 vs 시끄러운 실패** | 에러 없이 틀린 결과(사례 2) vs 에러 내고 죽음(사례 1·3~6). 조용한 쪽이 위험 — 기대값 비교로만 잡힘 |
| **고아 run** | 프로세스가 죽어 MLflow에 RUNNING으로 방치된 run. FAILED/KILLED로 정리 필요 (`with mlflow.start_run()` 안에서 예외로 죽으면 자동 FAILED) |
