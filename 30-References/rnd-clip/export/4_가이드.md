# 4. 가이드 — CLIP을 업무에 쓸 때의 판단 기준 (2026-07-10)

## 1. zero-shot / linear probe / 풀 파인튜닝 — 언제 무엇을

| 상황 | 선택 | 근거 (이번 실측) |
|---|---|---|
| 분류 체계가 자주 바뀜, 라벨 없음 | **zero-shot** | 프롬프트 교체 = 체계 교체 (5초, 재학습 0) |
| 라벨 소량(수십~수백), 체계 고정 | **linear probe** | 머리 1,026눈금·0.9초 학습, 오탐 억제형(P 0.889) |
| 라벨 대량 + 도메인 특수 | 풀 파인튜닝 검토 | 이번 R&D 범위 밖 (비용 대비 필요할 때만) |

**주의**: 잘 설계된 프롬프트의 zero-shot이 소량 probe와 동률일 수 있다(실측 0.7778 동률).
probe를 선택하기 전에 **프롬프트 개선 여지부터 소진할 것** — 라벨링보다 문구 수정이 싸다.

## 2. 프롬프트 설계 수칙 (실측 근거)

1. **부정형 금지** — "no people" 류는 동전 이하(0.4444). CLIP은 부정을 못 읽는다.
2. **대비쌍으로** — 찾는 것 vs "그 외"의 **긍정적 묘사** ("동물·음식·차량·사물 사진"). 0.7778.
3. **캐치올 클래스 포함** — 닫힌 집합이라 어느 후보에도 안 맞으면 억지 배정 (BERT 36.61% 교훈 동일).
4. **언어 일치** — 프롬프트 언어를 텍스트 인코더가 읽을 수 있어야 (영어 모델+한국어 = 0.5185 붕괴).
   한국어 업무는 KoCLIP (실측 0.8519 최고).

## 3. 분류 체계가 사는 곳 — 방식 선택의 본질

| 방식 | 체계의 위치 | 변경 비용 |
|---|---|---|
| 룰 | 코드의 조건문 | 코드 수정 |
| BERT류 | 가중치 (분류층+label_map) | **재학습** |
| CLIP zero-shot | **입력 문자열 (프롬프트)** | 문자열 교체 (즉시) |

체계가 자주 바뀌는 업무(새 위험 유형 추가 등)일수록 CLIP 쪽이 유리하고,
체계가 고정이고 정확도가 생명이면 학습형이 유리 — **변경 빈도가 방식을 고른다.**

## 4. UXUI(지시 3)에 싣는 확정 구성

- 이미지 탭 = **YOLO 박스(무엇이 어디에) + KoCLIP 상황 태그(전체 분위기가 무엇)** — 상호 보완:
  YOLO는 학습된 80클래스의 위치를, CLIP은 임의 프롬프트의 전체 맥락을.
- 상황 후보(한국어)는 운영하며 교체 가능 — 재학습 없이 (이게 CLIP을 고른 이유).
- 도박 도메인 프롬프트·실제 이미지 검증은 UXUI 단계의 몫 (coco128은 일상 사진뿐).

## 5. 신뢰도 읽기 (모든 분류기 공통, CLIP도 동일)

- softmax %는 **후보들 사이의 상대 확신**일 뿐 — 후보가 전부 틀려도 합은 100%.
- 낮은 1위 확률(예: 40~50%)은 "모르겠다" 신호 — 캐치올 클래스와 함께 임계값 운영 권장.
- test 27장 실측치는 잠정 — 자체 이미지가 모이면 같은 잣대(고정 test)로 재채점.
