# ML 용어 정리 — 판정의 두 세계와 학습 루프 단위 (2026-07-09)

> 2026-07-09 복습 세션에서 물어본 용어만 정리. 기초 정의(P·R 공식, 오탐/미탐/정탐, 3렌즈, mAP)는
> [[mlflow-practice/mlflow-terms-glossary]]가 정본 — 이 문서는 **그날 새로 잡힌 각도만** 누적한다.
> 코드 인용은 전부 `30-References/rnd-dataset-artifacts/export/train_text.py` 기준.
> 숫자 예시는 설명용 가상값으로 통일: **전체 100건, 실제 이상 20건, 이상판정 10건 → TP 8 / FP 2 / FN 12 / TN 78**.

## 1. "판정" 주어 고정 어휘 — 네 칸을 틀릴 수 없게 부르기

한국어 번역("틀리게 판단한 것", "놓친 것")은 문장마다 주어가 바뀌어 반드시 꼬인다.
해결: **주어를 "판정"으로 고정**하고, 앞에 판정 이름·뒤에 채점만 붙인다.

| 고정 어휘 | 영어 | 표준 용어 | 예시 건수 |
|---|---|---|---|
| **이상판정맞음** | TP | 진양성 | 8 |
| **이상판정틀림** | FP | **위양성**(僞=가짜) | 2 |
| **정상판정맞음** | TN | 진음성 | 78 |
| **정상판정틀림** | FN | **위음성** | 12 |

- 영어가 간단한 이유: FP·FN은 묘사가 아니라 **두 칸 조립식** — `[T/F 채점] + [P/N 판정 이름]`.
  의학·통계의 위양성/위음성이 이 구조를 그대로 보존한 유일한 한국어 (False=위(가짜), Positive=양성판정).
- 이 어휘의 힘: **틀린 문장을 만들 수조차 없다.** "이상판정 중에 놓친 것"은 문법적으로 조립 불가 —
  놓침(FN)은 정상판정 쪽 어휘로만 표현된다.
- **FN도 능동적 오판이다.** "판단해내지 못함"이 아니라 "**아니다라고 판단했고 그게 틀림**".
  모델은 모든 입력에 반드시 판정한다(닫힌 집합 — "해당 없음" 선택지 없음).

### 한국어 번역이 실패하는 세 가지 방식 (경계 목록)
1. "틀리게 판단한 것" → 채점(F)만 번역, 판정 방향(P/N) 누락 — FP·FN 구분 불가
2. "틀린 것을 판단해내지 못한 것" → 주어가 대상(잡을 것)으로 이동 + '판단 안 함'이라는 착각 유발
3. "오탐/놓침" → 결과의 별명으로는 유용하나 정의가 아님 — 정의를 말할 땐 위양성/위음성

## 2. 분모의 정체 — P·R·accuracy를 가르는 것은 채점 범위뿐

**분모는 언제나 "집단 전체"이고, FP·FN은 그 안의 틀린 조각일 뿐** (조각 자체가 분모가 아니다).

| 지표 | 한 문장 (같은 문형) | 분모 = 채점 범위 | 분자 | 예시 |
|---|---|---|---|---|
| **정확도** | 모든 판정이 얼마나 맞았나 | 전체 (네 칸 전부, 100) | 맞은 판정 전체 TP+TN (86) | 0.86 |
| **P (정밀도)** | 이상판정이 얼마나 안 틀렸나 | 이상판정 전체 = 적중+오판 (10) | TP (8) | 0.80 |
| **R (재현율)** | 실제 이상을 얼마나 안 놓쳤나 | 실제 이상 전체 = 적중+놓침 (20) | TP (8) | 0.40 |

- accuracy에서 TP+TN은 **분자**다 (분모로 착각 주의). 분모는 맞음·틀림을 다 포함한 전체 표본.
- 위 예시가 accuracy의 맹점을 그대로 보여줌: **acc 0.86인데 R 0.40** — 분자의 대부분(78/86)이
  다수 클래스의 TN이라서. 극단: 이상 1% 데이터에서 "전부 정상" 모델도 acc 0.99 (R은 0).
- 두 분모의 차이는 딱 하나 — **적중(TP)에 무엇을 더하느냐**: 오판(FP)을 더하면 판정의 세계(P의 분모),
  놓침(FN)을 더하면 현실의 세계(R의 분모).

## 3. 두 원(집합) 관점 — 이상판정 vs 실제 이상

**이상판정**과 **실제 이상**은 주인이 다른 별개의 집합이다. 이 둘이 다르다는 사실이 P와 R이 두 개 존재하는 이유 그 자체.

```
이상판정 (모델의 원, 10건):   [ FP 2 ][ TP 8 ]
실제 이상 (세상의 원, 20건):          [ TP 8 ][ FN 12 ]
                                      └─ 겹친 부분 ─┘
```

- **모델의 원**: 모델이 그림. 모델이 바뀌면 원도 바뀜. 모델만 돌리면 공짜로 얻음.
- **세상의 원**: 세상(정답)이 정함. 모델과 무관하게 불변. **라벨링해야만 존재** (= ground truth 제작).
  라벨링이 곧 "세상의 원을 그리는 일" — 데이터셋이 1급 자산인 이유의 평가 버전.
- **P = 겹침 ÷ 모델의 원** (8/10), **R = 겹침 ÷ 세상의 원** (8/20). 같은 분자를 다른 원으로 나눈 것.
- 학습의 목표 = 두 원을 포개는 것. 완전히 포개지면 P = R = 1.
- R은 세상의 원 없이는 계산 자체가 불가 — 놓친 것(FN)은 판정 목록에 안 보이므로.

## 4. 두 가족 — 같은 것의 다른 이름들

ML 용어는 같은 것을 여러 이름으로 부른다. 가족만 구분하면 절반은 끝난다.

| 가족 | 동의어들 | 코드에서 (train_text.py) | 누가 만드나 |
|---|---|---|---|
| **모델의 출력** | 판정 = 분류 = 예측(prediction) | `preds` (103행, `logits.argmax(-1)`) | 모델 |
| **정답** | 라벨 = 정답지 = gold = ground truth | `golds` (104행, `batch["labels"]`) | 사람 (라벨링) |

- **T/F는 두 가족을 한 건씩 맞대는 순간 태어난다**: `preds[i] == golds[i]` → T, 다르면 F.
  같은 가족끼리 비교하는 문장("분류한 것과 예측한 것을 비교")은 자기 자신과의 비교 — 성립 불가.
- 평가란 결국 **모델의 목록과 세상의 목록을 나란히 놓고 맞대보는 일**. 이 문서 전체가 이 한 문장의 주석이다.

## 5. 코드 대응표 — 개념이 사는 주소 (train_text.py)

| 개념 | 행 | 코드 |
|---|---|---|
| 판정(P/N) 탄생 | 103 | `preds += logits.argmax(-1)...` — 확률 최고 클래스 선택 |
| 정답지 수집 | 104 | `golds += batch["labels"]...` — gold = ground truth 관례 |
| **T/F 탄생** | 155·160-161 | `accuracy_score(golds, preds)` / `f1_score(...)` — 원소별 `==` 비교는 sklearn 내부 |
| 라벨 번호 부여 | 115-116 | `classes` 정렬 → `label2id` (문자열이든 숫자든 고정 번호) |
| labels 릴레이 | CSV→127→84→92 | 라벨 열 → 번호 목록 → 텐서 → 배치에 문장별 탑재 |
| 정답의 훈련 사용 | 147 | `model(**batch).loss` — **"labels" 키는 HF 계약**: 이 이름이면 loss까지 자동 계산 |

- **labels를 "정의"하는 곳은 코드에 없다** — 코드는 운반·변환만 한다. 값을 정한 것은 CSV에 라벨을
  붙인 사람 (= "데이터셋이 분류 기준의 정본"의 코드 증명).
- 네 칸(TP/FP/FN/TN)을 직접 보려면: `zip(preds, golds)`에서 `p == 1`(판정 이름)과 `p == g`(채점)의
  조합으로 센다 — 두 칸 문법이 조건식과 1:1. (현재 스크립트엔 없음 — 필요 시 추가)

## 6. 학습 루프의 단위들 — 포개기 한 판

실행 예시 기준: train 62문장, `BATCH=16`, `MAX_LEN=128`, `EPOCHS=1`.

```
학습 전체 (EPOCHS 바퀴)
 └─ 에폭 = 훈련 데이터 전체 한 바퀴 (1회독)
     └─ 스텝 = 배치 1개 처리 + 가중치 수정 1회 (에폭당 4스텝)
         └─ 배치 = 한꺼번에 모델에 들어가는 묶음 = [16문장 × 128토큰] 직사각형 텐서
             └─ 문장(건) = 학습의 낱개 단위 (텍스트분류 1건=문장, YOLO 1건=이미지)
                 └─ 토큰 = 문장 내부 단위 (max_len이 상한)
```

| 단위 | 값 | 누가 정하나 |
|---|---|---|
| max_len (토큰 상한) | 128 | 사용자 |
| 총 건수 | 62 (**train 분할** — CSV 전체 90이 아님) | 데이터 |
| 배치 크기 | 16 | 사용자 |
| 스텝 수/에폭 | 4 | **자동** = ⌈62÷16⌉ |
| 에폭 수 | 1 | 사용자 |

핵심 공식과 성질:
- **스텝 수(에폭당) = ⌈총 건수 ÷ 배치 크기⌉** — 나머지는 올림: 자투리 14건도 작은 배치로 1스텝 (버리지 않음).
- **총 스텝 = 에폭당 스텝 × EPOCHS** = 가중치 수정 총횟수. "배운다"의 실체가 이 횟수.
  스텝당 시간 × 총 스텝 = **학습 시간 예측**에 바로 쓰임.
- **배치가 존재하는 이유 둘**: ① GPU는 표를 통째로 곱한다 — 16문장이 줄서기가 아니라 **동시에** 계산됨
  (batch = 빵 "한 판" 굽기 = 일괄). ② 16건 오차의 **평균**으로 수정 → 한 문장의 변덕에 안 흔들림.
- **배치는 추론에도 있다**: 훈련의 배치 = 가중치 수정 1회분 재료, 추론의 배치 = 한꺼번에 처리할 일감.
- **max_len은 문장의 속성이 아니라 사용자가 강제하는 상한**: 넘치면 자름(truncation — 뒷부분 정보 소실),
  모자라면 채움(padding). 배치를 직사각형으로 만들기 위한 규격화. 토큰 ≠ 글자·단어 (서브워드 분해).
- **에폭 ↑ = 회독 ↑**: 실측 — YOLO 데모 5에폭 mAP50 0.06 → 40에폭 0.87. 단 과하면 답 암기(과적합) —
  에폭마다 찍히는 검증 점수가 꺾이는 지점이 멈출 신호 (validation의 존재 이유).
