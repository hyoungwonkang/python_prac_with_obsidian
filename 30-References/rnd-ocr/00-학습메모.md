## OCR의 구조

### 1. OCR = 검출 + 인식, 2단 파이프라인
- 출력이 3요소인 이유: `(박스 좌표, 텍스트, 신뢰도)` — **검출(글자가 어디에) + 인식(그 글자가
  무엇인지)**이 한 몸이라 위치와 텍스트가 함께 나온다. YOLO(어디)+분류(무엇)의 결합판.
- EasyOCR 내부: 검출=CRAFT, 인식=CRNN (둘 다 PyTorch 사전학습 — 우리 쪽 **학습 0줄, 추론만**).
  성격은 PII(ko-pii)·OpenCV와 같은 라이브러리 실습, 아키텍처 ②추출 계층.

### 2. 깨끗한 이미지도 오독한다 — OCR은 이해가 아니라 모양 추정
- 첫 실측(clean, 완벽 렌더링): CER 9.5%, 4글자 오독 — **전부 모양이 비슷한 글자끼리**
  (첨→점, 을→올, 1→/, 랍→람). 획 하나 차이는 구조적 약점.
- 신뢰도가 자기 오독을 표시: 오독 2곳 줄이 0.325로 스스로 낮게 채점 → 검수 라우팅의 근거.
- 하류 파이프라인 조짐: 스팸 어휘("무료 상품권")는 생존 → 분류는 견딜 가능성 /
  전화번호가 `0/0-…`로 깨짐 → **PII 정규식은 놓칠 수 있음** — OCR 오류가 하류 탐지를
  무너뜨리는 경로의 실물.

## 채점 잣대

### 3. CER = 편집거리 ÷ 정답 글자 수 — 부분 점수의 잣대
- 받아쓰기 채점: "정답 글자 100개당 몇 개 틀렸나". 0%=완벽, 낮을수록 좋음(F1과 방향 반대),
  100% 초과 가능. 한국어는 교착어라 단어 단위(WER)가 아닌 **글자 단위(CER)가 표준**.
- 지표 지도에서의 자리: accuracy·F1(건 단위 O/X)·mAP(박스)와 달리 CER은 문자열이
  **얼마나** 틀렸는지의 부분 점수 — "거의 맞음(9.5%)"과 "붕괴(64.3%)"를 한 숫자로 가름.

### 4. 편집거리 = 가장 짧은 고치기 경로 — 밀림은 길이 변화형만
- 최소 수정(치환·빠짐·끼어듦) 횟수. **"최소"가 핵심** — 같은 두 문자열도 서툰 길(3회)과
  영리한 길(1회)이 있고, 편집거리는 항상 후자. DP 표 채우기로 계산(ocr_eval.py:22-33), 대칭.
- 자리별 비교(눈 대조)가 무너지는 원인은 삭제가 아니라 **길이 변화**(빠짐·끼어듦) — 뒤 전체가
  밀림. 치환은 길이 불변이라 안 밀림. "빠짐=삭제냐 삽입이냐"는 변환 방향이 붙이는 이름표일 뿐.
- 눈 대조의 한계 실증: 같은 오류를 나는 "4곳", 코드는 "3구간(4글자)"로 셈 — 사람은 세는
  기준이 흔들린다. clean에서 눈 대조가 통한 건 오류가 마침 치환뿐이었던 우연.

### 5. 돋보기와 잣대의 분리 — 지시 2 패턴 재사용
- `ocr_easy.py`(돋보기) = 정답 모름, 신뢰도·박스까지 표시 → 오답 **원인** 분석용 /
  `ocr_eval.py`(잣대) = 정답 대조 CER, `make_reader()`로 **엔진 중립** → 점수는 여기서만.
- rule_spam.py ↔ eval_compare.py 구조 그대로 — 잣대가 특정 엔진 안에 있으면 "각자 자기
  자로 잰" 꼴. easy|paddle을 같은 자로 재기 위한 분리.

## 실험 설계

### 6. 열화 4종 = 의도적 파괴 — 전처리 가치 증명의 대조군
- 열화는 실무 저품질의 표본이자 **OpenCV 전처리 4종의 시험지**: 회전→기울기 보정 /
  노이즈→디노이즈 / 저대비→이진화 / 축소→확대. 망가진 기준선 없이는 "전처리가 살렸다"를
  증명할 수 없다 (CLIP 쇼핑몰 대조 실험과 같은 구조).
- 기준선 실측(EasyOCR·CPU): clean 9.5 / lowcontrast 7.1 / noisy 7.1 / **rotated 64.3** / small 9.5(%).

### 7. rotated 64.3%의 실체 = 어순 붕괴 — 그리고 비단조 관찰
- 글자를 못 읽은 게 아니라 **검출 박스의 줄 묶음·정렬이 꼬여 순서가 붕괴** → 빠짐·끼어듦
  대량 → 편집거리 폭발. 기울기 보정이 정확히 이걸 고치는 전처리 (다음 실험 1순위).
- 예상 밖: lowcontrast·noisy가 clean보다 좋음(7.1 < 9.5) — 열화로 `1→/` 오독이 사라짐.
  **화질과 인식은 단조 관계가 아니다.** 단 문장 1개(n=1) 결과 — 일반화 금물.

## 운용 원칙

### 8. 장치 보수주의 — 이득 0이면 위험도 사지 마라
- OCR 채점은 CPU 기본(GPU=1 옵션만). 근거: ① MPS 조용한 오답 전례
  ([[../../10-Projects/llm-from-scratch/llm-ch7-failure-log]] — 비연속 가중치+MPS 행렬곱,
  같은 배치 CPU 3.81 vs MPS 19.77) ② 추론 0.1~0.2초/장이라 GPU 이득이 사실상 0
  ③ ocr_eval은 **잣대** — 장치 따라 흔들리면 모든 비교가 오염. 장치 고정 = seed 고정과
  같은 변수 통제. (학습처럼 속도가 아쉬울 땐 MPS + 수치 검증으로 상쇄 — 국면별 계산.)

### 9. 신뢰도 = 인식 모델의 softmax 확률 — 표준 출력, 같은 족보
- readtext 3요소의 신뢰도는 EasyOCR 전용 기능이 아니라 확률 기반 엔진의 표준 부산물
  (PaddleOCR은 rec_scores). BERT %·YOLO 0.92·CLIP %와 같은 "모델의 자기 확신".
- 자기 확신 ≠ 정답 보증: 모양이 아주 비슷한 오독은 **높은 신뢰도로 당당히 틀릴 수 있다**
  (OCR판 확신형 — [[../rnd-clip/00-학습메모]] 억지 배정 유형론의 재현).

### 10. CER ↔ 신뢰도 = 국면 릴레이 — 그리고 검수는 사람이다
- **평가 국면**(정답 있음): CER 주역, 신뢰도는 참고 / **운영 국면**(정답 없음 — 알았다면
  OCR이 불필요): CER 계산 불가, **신뢰도가 유일한 품질 신호**. R&D의 숙제 = CER로 신뢰도
  문턱을 검증해(validation 정신) 운영 라우팅에 인계.
- 검수 = 사람의 직접 확인·수정. 신뢰도는 **사람 시간의 배분기**(확신 높음→자동 통과,
  낮음→검수 큐) — 50% 경고·의사 라벨링과 동일 패턴이고, 고쳐진 쌍은 골드 데이터로 축적돼
  재학습 순환에 편입. 사람 앞에 자동 재시도 사다리(전처리 재OCR→타 엔진→사람) 가능.
- 이 도메인 특수성: 검수량 축소가 곧 **검수자 보호**(유해물 노출 최소화), 검수 화면은 비식별본.
