## NER — 토큰 분류의 구조

- **문장 분류 vs 토큰 분류**: 같은 BERT 본체에 다른 머리. 스팸=문장당 라벨 1개([CLS]), NER=**토큰마다 라벨 1개**(13칸 점수 벡터→argmax). "인명이 있나(유무)"가 아니라 "**어느 글자가** 인명인가(위치)"를 짚으려면 토큰 분류여야 함.
- **정답지 vs 답안지**: `batch["labels"]`=사람이 단 정답(고정), 모델 출력=계산된 추측(학습하며 변함). 모양이 같은 이유는 대조(채점)하려면 형식이 같아야 하기 때문. 학습=답안지를 정답지에 가깝게, 평가=둘을 대조해 F1.
- **정답 라벨의 물리적 위치**: KLUE-NER은 `~/.cache/huggingface/datasets/klue___klue`(37MB)의 **`ner_tags` 필드** — KLUE 프로젝트 주석자들이 글자마다 수작업 태깅한 숫자들. 데이터셋의 가치가 곧 이 수작업 라벨.
- **라벨 13종 = 6유형(PS·LC·OG·DT·TI·QT) × B/I + O**. B(개체 시작)/I(이어짐)를 나누는 이유: 개체 둘이 붙어 있을 때 경계 구분. O=개체 아님.
- **`Auto...` vs `Bert...`**: `AutoModelForTokenClassification`=자동 선택기(config의 model_type을 보고 실물을 고름), `BertForTokenClassification`=transformers 라이브러리에 정의된 실물(bert이므로 이것이 생성됨). 스팸 코드는 실물 직접 지명, NER 코드는 Auto(MODEL_NAME 교체 유연성).

## subword와 토크나이저 (실측으로 확인)

- **subword = 토크나이저가 단어를 쪼갠 조각** — 단어보다 작고 글자보다 큰 중간 단위. 2장 BPE와 같은 개념(BERT는 WordPiece). `##`=앞 조각에 붙어있던 꼬리.
- **토크나이저 ≠ BERT 본체**: 토크나이저=신경망 아닌 전처리 부품(어휘사전+자르기 규칙), 본체=신경망. 어휘사전 칸 번호가 계약이라 항상 세트로 사용. klue/bert-base도 실물 파일 있음(`~/.cache/huggingface/hub/`에 model.safetensors 425MB+config+vocab) — HF는 주소로 부르고 캐시에 숨기는 관례, ultralytics는 파일명으로 부르고 작업폴더에 두는 관례.
- **토크나이저는 의미가 아니라 어휘 통계로 쪼갬** (klue 실측): `홍길동`→1토큰(유명해서 사전에 통째로), `김철수를`→`김철`+`##수`+`##를`(김철수는 사전에 없음), `삼성전자`→1토큰. **조각 경계는 개체 경계와 무관** — `김철`이 첫 조각인 건 이름 경계가 아니라 사전 사정.
- **KLUE-NER이 글자 단위 라벨인 이유**: 한국어는 교착어라 개체 경계가 어절 내부에 있음(`김철수|를`) — 어절 단위론 개체와 조사를 못 가름. 글자 단위면 김/철/수=PS, 를=O로 정확히 분리.
- **-100 정렬의 실제 발동 위치**: 글자 단위(우리 방식)에선 한글 글자≈1토큰이라 **-100은 주로 [CLS]/[SEP]/패딩에만** 붙음. "첫 subword만 라벨, 나머지 -100"은 단어 단위 입력(`홍길동은`→`홍길동`[B-PS]+`##은`[-100])이나 숫자·영문 쪼개짐 때 발동하는 안전장치.
- **개체 이어짐 ≠ subword 이어짐** (층위 구분): `길`·`동`의 I-PS는 **진짜 라벨**(개체가 계속됨), -100은 **한 글자가 여러 조각으로 쪼개질 때 둘째 조각부터**(채점 제외). 완전히 다른 것.

## YOLO — 객체 탐지의 구조

- **박스(bounding box) = 물체를 둘러싼 네모**, 위치 = `중심x·중심y·너비·높이` 4수(이미지 대비 0~1 비율 — 리사이즈에도 유효). NER이 글자 구간을 짚듯 YOLO는 픽셀 영역을 짚음.
- **jpg=문제지, txt=정답지**: `datasets/coco128/images/*.jpg`(사진)와 `labels/*.txt`(한 줄=박스 하나=`클래스 x y w h`)가 같은 파일명 쌍. txt가 COCO 주석자들의 수작업 정답 — KLUE `ner_tags`의 이미지판.
- **비율→픽셀 산술**: `0.479×640=306.9`(중심), 중심±크기/2로 모서리 좌표. 실측: 음식사진의 bowl 박스 = 가로 거의 전체×세로 아래 60%.
- **신뢰도(conf) = 모델이 출력하는 자기 확신** — 후보 박스마다 `[좌표4 + 80칸 클래스 점수]`를 내고, 로짓→시그모이드(ultralytics `head.py:184`)→최고 칸의 (번호=클래스, 값=신뢰도). 스팸 `predict_spam.py:50`의 `F.softmax` 확신도와 같은 계보 — 라이브러리 속으로 들어갔을 뿐. **칸 계보: GPT-2 50,257(어휘) → 스팸 2 → NER 13(태그) → YOLO 80(COCO 클래스)**.
- **예측 박스의 개수**: 내부 후보 이미지당 8,400(격자 고정) → conf<0.25 필터+NMS(중복 제거) → 살아남은 것만 `results`에 (실측: 음식사진 6개, 박스당 `[x1,y1,x2,y2,conf,cls]` 6수로 압축 — 80칸 벡터는 도착 전에 요약·폐기됨).
- **conf vs IoU는 별개 축**: conf="무엇인지 얼마나 확신"(모델의 주관, 추론 때도 존재), IoU="어디가 정답과 겹치나"(객관, 정답 박스 있어야 계산). TP = 클래스 일치(①) **그리고** IoU≥합격선(②) — conf는 판정에 직접 안 쓰이고 필터(0.25)·정렬(AP)용.
- **train/predict 분담**: `yolo_train.py`=학습(가중치 바뀜, loss→backward→step), `yolo_predict.py`=추론만(가중치 읽기). llm_finetuning/save_response·finetune_ner/predict_ner과 동일 구조.

## PII — ko-pii의 구조 (2026-07-06 Q&A)

- **Anonymizer = 익명화 기계 클래스** (ko-pii의 대표 실물 — transformers의 BertFor...에 해당). 내부 3부: **탐지부**(룰·사전·체크섬) + **처리부**(strategy대로 치환) + **보관부**(Vault). 손잡이 둘: `mode`(탐지 엄격도, STRICT 등 5단계) · `strategy`(가리기 방식, tokenize=복원 가능 번호표).
- **탐지 3장치의 분업**: 형식 있는 것(전화·이메일)=**룰(정규식)**, 형식·수학 없는 것(인명·주소)=**사전+휴리스틱**(어림 규칙 — 보장 없는 경험적 추측), 검증 수학 있는 것(주민번호·카드·사업자)=**체크섬**. 실측 대비: 홍길동(문맥어 "신청인") conf 1.00 / 김철수("고객님") 0.55 REVIEW / 김영희 미탐 — **휴리스틱의 흔들림이 곧 룰 기반 인명탐지의 태생적 한계** → NER 하이브리드 권장의 실증 근거.
- **주민번호 체크섬 산술**: 앞 12자리 × 가중치(2~9,2~5) 합 → mod 11 → 11에서 뺌 = 끝자리여야 함. `880101-123456`→146→나머지3→**8**=끝자리 ✓ (그래서 conf 1.00). 한 자리만 틀려도 불통과 → 형식 흉내 가짜(주문번호 등)를 걸러 **오탐(FP) 급감** — 대신 **2020-10 이후 신규 발급분은 뒷자리 임의번호라 검증식 불성립** → "체크섬 실패=가짜" 단정 금지(미탐 위험). FP↓의 대가로 FN 위험을 얹는 정밀도↔재현율 트레이드오프의 재현.
- **형식 검증 vs 본인확인은 다른 층**: 체크섬=오프라인 산수로 "형식상 유효한가"(오타 검출·데이터 정제·PII 탐지용 — 현실 표준), 실존 인물 확인=공적 조회(NICE·KCB 등) 필요. 우리 용도(탐지·마스킹)엔 체크섬이 정확히 맞는 도구.
- **오탐·미탐 한자 독법**: 오탐=誤探 — 오(誤)=False(채점: 틀림)+탐(探)=Positive(대답: 잡았다), 글자 대 글자 대응. 미탐=未探="탐지 못 함" **통짜** 대응(미=F, 탐=N 아님). TP·TN은 한국어 단어가 따로 없음.
- **번호표(`<RRN_1>`) 설계 = 유형 + 순번**: **유형**(RRN·PERSON)을 남기는 이유 — 개인 특정은 못 하면서 위험 판정(RRN 등장→CRITICAL)·문장 독해·정책 집행("RRN 포함 문서 등급 상향")·집계가 가능해짐. **순번**(_1)= 같은 유형 내 개체 구별 + 재등장 시 동일성 유지(`<PERSON_1>` 재등장=같은 인물 — 관계 구조 보존) + Vault 복원 열쇠. ⚠️ RRN 명칭 속 '번호'(원본 값 그 자체)와 _1(관리용 순번)은 전혀 다른 층.
- **tokenize가 redact(■■■)·mask(***)보다 나은 이유**: 문장이 읽히고 유형·관계가 살아서 **마스킹 상태로 후속 처리**(검수자 독해·LLM 분석·RAG) 가능 + Vault로 권한 시 복원 — 보호와 분석의 양립.
- **Rule Engine(자리) vs ko-pii(부품)**: 아키텍처의 Rule Engine은 설계도의 계층 이름(PII 룰 + 도메인 위험 키워드·은어 룰까지), ko-pii는 그중 **PII 조각을 실증한 기성 부품 후보**. 도메인 룰은 별도 과제([[../../10-Projects/detection-ai-study]] 2군 RULE), 실제 채택은 별도 기술 선정 결정 — R&D는 그 근거 자료.

## 전이 사슬 A/B 실측 — WikiAnn(ko) 도메인 확장 (2026-07-07)

- **실험 설계**: 같은 데이터(WikiAnn ko 8,000건)·같은 조건에서 **출발점만** 바꿔 비교 — A=klue/bert-base 원본, B=ner_klue.pt의 본체 이식(분류층 13칸은 체계 불일치로 제외, 7칸 신규). 코드 `finetune_wikiann.py` (INIT_FROM 환경변수).
- **결과**: A **0.8447** vs B 0.8317 — **전이 2회차가 근소하게 열세** (전 클래스 일관). 결론은 "B가 나쁘다"가 아니라 "**사슬 연결의 추가 이득이 확인되지 않음**" (시드 1개, 차이 0.013은 노이즈 가능).
- **해석**: ① 파인튜닝 산출물의 본체는 이전 과제(뉴스·글자 단위)에 특화된 상태 — 다른 분포(위키·어절 단위)에선 편향으로 작용 가능 ② 하류 데이터가 충분하면(8k×2ep 수렴) 출발점 이점이 씻김. **실무 기본값: 새 과제는 범용 사전학습 본체에서 새로 파인튜닝** — "계속 학습=전이"의 사슬이 항상 이득은 아님.
- **부수 확인**: 어절 단위 데이터에서 F1 0.84 = **"첫 subword만 라벨, 나머지 -100" 정렬의 실전 데뷔 성공** (KLUE 글자 단위에선 잠자던 규칙). WikiAnn은 silver(자동 생성) 라벨 — ORG 최약(0.75)은 그 소음+조사 혼입(`FC에서`[I-ORG]) 영향으로 추정. WikiAnn은 **test에도 라벨 있음** (KLUE와 달리 리더보드용 아님).
- **부분 로드 기법**: 모양이 다른 층을 건너뛰는 이식 = `state_dict`에서 해당 키 제외 후 `load_state_dict(body, strict=False)` — 전이 사슬·모델 개조의 기본 수법.

## 공통 패턴 (추상화 사다리)

- **loss→역전파→눈금조정의 노출 수준**: 교재=전부 직접(`train_model_simple`) → HF/NER=루프는 내가, loss는 모델이(`finetune_ner.py:95-100`에 세 단계 노출) → ultralytics/YOLO=`model.train()` 한 줄(실물은 라이브러리 `trainer.py:468 backward·495 step`에 은닉). **바닥부터 배운 가치 = 한 줄 안에서 도는 것을 앎.**
- **자동 다운로드**: "없으면 받고 있으면 쓰는" 패턴. 7장=urllib 직접 → KLUE=`load_dataset()` 대행 → COCO128=`"coco128.yaml"` 문자열만. 덕분에 "git엔 코드만, 데이터·가중치는 재생성" 원칙 성립.
- **전이학습 = 사전학습(범용, 남이, 비쌈) + 파인튜닝(내 작업, 소량, 쌈)의 2단계 전략** — 트랜스포머(구조)와 무관한 학습 전략이라 CNN 계열 YOLO에도 그대로. 지금까지 4회 반복: GPT-2 분류(6장)·지시(7장)·BERT 스팸→NER·YOLO. 능력은 1단계가 주입, 2단계는 꺼내 쓰는 법.
- **수치+실물 이중 검증**: MLflow run이 학습 시점 수치를 정확히 재현(F1 0.7057)=".pt 저장→로드 왕복 무손상"의 증거. 추론 눈확인(홍길동[PS]·bowl 0.92)까지 — MPS 조용한 버그의 교훈 계승.
- **.pt의 제조 공정 3단계** (ner_klue.pt = 상사 콘솔 이슈의 원인 이해): ① 재료 — 사전학습 가중치 로드 + 분류층 난수 초기화(LOAD REPORT의 MISSING) ② 변형 — 학습 루프가 눈금 조정(메모리에만 존재) ③ 저장 — **마지막 줄** `torch.save(model.state_dict(), ...)`. **state_dict = "층 이름 → 눈금 텐서"의 사전** — 조정된 눈금 전부에 이름표를 붙여 꺼낸 것. 완주 전에 멈추면 파일이 안 생김(상사 윈도우 건). 크기 = 파라미터 수 × 4바이트(fp32).
- **torch는 같은 버전명, 다른 빌드**: Windows에서 `pip install torch==2.12.1` 기본 = **CPU 전용 빌드**(GPU 있어도 무용) — CUDA 빌드는 `--index-url .../whl/cu124`로 별도 창고에서. 코드의 device 폴백(cuda→mps→cpu)은 준비돼 있어 설치만 바꾸면 자동 GPU. requirements.txt는 창고(index)를 못 적으므로 README에 분리 — **"의존성은 버전만이 아니라 빌드도"**.
