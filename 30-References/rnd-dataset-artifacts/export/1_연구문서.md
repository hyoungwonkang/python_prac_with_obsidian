# 1. 연구문서 — 학습 데이터 산출물 환경 구성 (확장성) (2026-07-08)

## 1. 배경과 목적

지금까지의 R&D(BERT 분류·NER·YOLO·PII)는 **과제마다 스크립트를 복사해 수정**하는 방식이었다.
데이터가 바뀔 때마다 코드를 고쳐야 하므로, 새 도메인(예: 불법도박·의심정황 텍스트, 현장 이미지)으로
확장할 때 반복 비용이 커진다.

이번 작업의 목적: **"새 데이터가 와도 규약만 맞추면, 코드 수정 없이
학습 → 산출물 → 재사용(분류)이 반복되는 환경"**을 만든다. (지시 항목 1 — 학습 데이터 산출물, 확장성 고려)

## 2. 핵심 개념 — 데이터셋과 산출물의 순환

```
학습 데이터 산출물(데이터셋 = 분류 기준의 정본)
        │  학습 (기술별 범용 학습기)
        ▼
모델 산출물 3종 세트(가중치+라벨맵+메타 = 기준의 실행형)
        │  추론 (재사용)
        ▼
미분류 새 데이터의 분류 결과 ──검수──▶ 데이터셋에 편입 (기준이 성장)
```

- **데이터셋이 1급 자산**: 가중치는 데이터+코드로 언제든 재생성 가능하지만 그 반대는 불가.
  특히 직접 라벨링한 데이터는 재생성 불가 → git으로 보존(`datasets-own/`).
- **고정 test셋 = 기존의 지표**: 같은 test셋으로 미래의 어떤 모델이든 동일 잣대로 비교한다.

## 3. 구성 요소 4가지

| # | 구성 요소 | 내용 |
|---|---|---|
| 1 | **텍스트 분류 범용 학습기** | `text,label` CSV(또는 3분할 디렉터리)만 주면 열·클래스 수 자동 인식 → 학습 → 산출물 세트. 새 과제 = CSV 준비가 전부 |
| 2 | **YOLO 커스텀 데이터 규약** | `images/ + labels/ + data.yaml` 폴더 계약 + 뼈대 생성기 + 범용 학습기. 직접 라벨링(지시 4번)이 이 규약 위에 바로 올라감 |
| 3 | **PII 분류 체계 정본 + 라벨 검증** | ko-pii 내장 33라벨을 코드에서 자동 추출한 스키마 문서(`pii_label_schema.md/json`) + 평가 데이터의 라벨을 체계에 강제하는 검증기 |
| 4 | **산출물 저장 규칙** | `artifacts/<이름>/`에 3종 세트(가중치·label_map.json·meta.json) 고정. 데이터는 공개(`datasets/`, git 제외)와 자체 제작(`datasets-own/`, git 추적)으로 분리 |

## 4. 검증 결과 (전부 로컬 실측, 2026-07-08)

| 검증 항목 | 데이터 | 결과 |
|---|---|---|
| 문자열 라벨·다중 클래스 자동 인식 | 합성 3클래스(경제/광고/스포츠) 90건 | 클래스 자동 인식, 정확도 1.00 |
| 기존 데이터 재사용 (디렉터리 계약) | 한국어 스팸 3분할, train 300건·1에폭 | 정확도 **0.9367** / 매크로F1 0.9367 |
| 산출물 재사용 분류 | `ko-spam-smoke` 세트 로드 | 스팸→`1`(59%), 정상→`0`(85%) 판정 |
| YOLO 규약 end-to-end | 합성 원/사각형 22장, 40에폭 | **mAP50 0.8718** (circle 0.995 / square 0.749) |
| PII 분류 체계 추출 | ko-pii 1.15.2 | **33라벨/5그룹** 스키마 생성, 평가 데이터 커버리지 10/33 |
| PII 라벨 검증 | 정상 샘플 + 오타 샘플 | 정상 통과(0) / 오타 `ACOUNT` 차단(1) |

모든 실험은 MLflow(sqlite)에 기록: 실험 `텍스트분류-범용`, `YOLO-커스텀`.

## 5. 확장 시나리오 (이 환경이 실제로 어떻게 쓰이나)

1. **새 텍스트 분류 과제** (예: 도박 권유/일반 대화):
   문자열 라벨로 CSV 작성 → `DATA=... NAME=... train_text.py` → 분류기 완성.
   기존 클래스 체계가 바뀌어도 코드 수정 없음 (클래스 수 자동).
2. **직접 라벨링한 이미지 학습** (지시 4번):
   `make_yolo_dataset.py`로 뼈대 → labelImg/Roboflow에서 YOLO 형식으로 라벨링 →
   `datasets-own/`에 배치 → `yolo_train_custom.py` 학습.
3. **PII 평가 데이터 확장**: 미커버 23개 라벨의 샘플 문장을 추가 → `validate_pii_samples.py`가
   라벨 오타를 차단 → `make_pii_schema.py` 재실행으로 커버리지 갱신.
4. **전이 확장**: 기존 산출물 가중치를 새 과제의 초기값으로 사용(NER R&D에서 검증한
   전이 사슬 방식) — 산출물이 쌓일수록 다음 학습이 좋아진다.

## 6. 한계와 다음 단계

- 텍스트 학습기는 **문장 단위 분류 전용** — NER(토큰 태깅)은 데이터 형식이 달라 별도 범용화 필요
  (기존 `finetune_ner.py`의 일반화, 동일 패턴).
- PII 평가 데이터는 33라벨 중 10개 커버 — 스키마 문서의 ▫️ 라벨부터 샘플 확장.
- "분류 잘하는 법"(지시 2번) = 이 환경의 고정 test셋 + MLflow 비교 위에서 진행할 다음 주제.
