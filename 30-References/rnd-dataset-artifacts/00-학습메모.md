## 이번에 새로 배운 것 (누적 원칙 — 이전 R&D에 없던 것만)

### 1. "학습 데이터 산출물"의 개념 정리 — 용어 3층
대화로 확정한 구분. 혼용하면 대화가 어긋난다.
- **학습 데이터 산출물** = 데이터셋 자체 (train.csv, 라벨링 이미지, PII gold set) — **분류 기준의 정본**
- **모델 산출물** = 가중치 세트 (.pt + label_map + meta) — 기준의 **실행형**
- 확장성 = 새 미분류 데이터를 ① 기존 산출물로 분류(추론)하거나 ② 기준 자체를 성장(재학습·편입)시키는 순환

### 2. 산출물은 "3종 세트"여야 재사용 가능
.pt 단독은 반쪽 — **label_map.json**(숫자↔클래스명) 없으면 출력을 해석 못 하고,
**meta.json**(베이스모델·max_len) 없으면 같은 구조를 재구성 못 한다.
기존 R&D 스크립트들은 라벨 매핑이 코드에 하드코딩돼 있었다 → 이번에 파일로 분리.

### 3. 계약(Contract) 사고방식 — 확장성의 실체
"코드를 데이터에 맞추는" 대신 "데이터가 지킬 모양(계약)을 정하고 코드는 고정":
- 텍스트: `text,label` CSV → 열·클래스 자동 인식 (2클래스든 7클래스든 코드 동일)
- YOLO: `images/ + labels/ + data.yaml` 폴더 → labelImg·Roboflow 출력이 그대로 들어옴
- PII: `문장<TAB>라벨` + 라벨은 33체계 강제 (검증기가 게이트)

### 4. 분류 체계는 라이브러리에서 "추출"한다 (손으로 베끼지 않음)
ko-pii의 33라벨이 `ko_pii.labels.ALL_LABELS`·`LABEL_INFO`로 코드에 존재 →
`make_pii_schema.py`가 자동 추출해 스키마 문서 생성. 라이브러리 업그레이드 시 재실행으로 갱신.
손으로 베낀 문서는 반드시 낡는다.

### 5. MLflow 3.x 파일 저장소 폐지 (실전 함정)
`./mlruns` 파일 저장소는 유지보수 모드 — sqlite 요구. ultralytics 학습 뒤에는
프로세스 내 추적 URI가 바뀔 수 있어 **명시 고정**(`set_tracking_uri`)이 필수였다.
같은 스크립트가 단독으론 되다가 YOLO 뒤에서 죽는 형태로 나타남.

### 6. 데이터의 git 이분법
- 공개·합성 데이터(`datasets/`) = 재생성 가능 → git 제외
- **직접 라벨링 데이터(`datasets-own/`) = 재생성 불가 자산 → git 추적**
- `ko_spam_raw.csv`를 자체 제작으로 오인했다가 정정한 것이 계기 (HF 다운로드본이었음)

## 직접 실행하며 배운 것 (2026-07-08 실습 세션 — 누적, 위와 중복 없음)

### 7. "3분할" ≠ 열 3개 — 숫자 3이 붙는 곳이 다르다
- **3분할 = 파일 3개**: train.csv(문제집) / validation.csv(모의고사) / test.csv(수능)
- **열은 2개뿐**: text + label. CSV 맨 앞의 `0,`·`1,`은 행 번호가 아니라 **라벨 값**
- test셋을 학습에 안 쓰고 아껴두는 이유 = 미래 모델과의 공정 비교 (고정 잣대)

### 8. "자동 인식"의 정확한 범위 — 파일 지정은 사람 몫
- 스크립트는 데이터 파일을 찾아주지 않는다. `DATA=` 환경변수로 지정 (미지정 시
  train_text.py:33 기본값인 ko 스팸으로 돌아감)
- 자동인 것은 **지정된 파일 안의 해석**: 열 이름(대소문자 무관)·클래스 수·문자열 라벨 매핑
- `KEY=value 명령` 접두 문법 = 그 실행 한 번에만 적용되는 환경변수

### 9. LOAD REPORT의 UNEXPECTED/MISSING은 파인튜닝의 정상 풍경
- 파인튜닝 = **몸통(사전학습 가중치) 재사용 + 머리(classifier) 새로 달기**
- UNEXPECTED = 베이스의 사전학습용 부품(MLM·NSP 헤드)을 버렸다 / MISSING = 새 분류 머리를 백지 초기화했다
- 재사용(classify_text.py:39-41) 때도 같은 리포트가 뜨지만, 직후 `load_state_dict`가
  model.pt로 **전부 덮어쓰므로** 중간 조립 단계의 메시지일 뿐

### 10. 라벨만 보지 말고 확률을 같이 읽는다 — 닫힌 집합의 한계
- 3클래스 랜덤 찍기 = 33.3%. 실측 36.61%는 사실상 "모르겠다"
- 닫힌 집합 분류기에는 **"해당 없음" 선택지가 없다** — 세계 밖 문장도 억지로 하나를 고름
- → 실제 과제에는 **"일반" 클래스가 필수** (도박 탐지를 "권유/일반 대화" 2클래스로 설계한 근거)

### 11. 정확도 1.0의 올바른 해석 — "배관이 뚫렸다"
- 합성 toy 데이터는 클래스별 어휘가 확연히 달라 일부러 쉬움 → 1.0은 파이프라인 검증이지 모델 품질 아님
- 같은 모델도 학습 분포 밖 문장에선 확신이 53%/36%로 추락 (test 정확도와 실전 성능은 별개)

### 12. 가상환경 역할 구분 — rnd-env vs ml-env
- **rnd-env** (Py 3.12, torch 2.12·transformers 5.x·ultralytics·ko-pii) = 탐지 R&D용. 이 환경 전용
- **ml-env** (Py 3.9, torch 2.8·transformers 4.x) = 기존 학습 프로젝트용 (llm-from-scratch 등)
- 산출물은 **만든 환경과 같은 환경에서 재사용** (5.x로 저장한 model.pt를 4.x에서 로드 위험)
- ⚠️ 이 구분이 정본 [[../pytorch-env-hybrid]]에 아직 미기재 — 갱신 후보

### 13. 세션 scratchpad는 휘발된다 — 데이터의 정위치
- 세션 임시 폴더에 만든 toy3.csv가 세션 종료 후 소멸 (meta.json에 죽은 경로만 남음)
- 합성 데이터도 `datasets/`에 둬야 로컬 보존됨 (git 제외여도 재생성 스크립트·규약이 있으면 충분)
