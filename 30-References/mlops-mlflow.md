## 1. "높은 레벨"이란 무엇인가 — 추상화 수준

영어 **"at a high level"**의 직역으로, **세부 구현은 빼고 전체를 멀리서 보는 시점**을 뜻합니다. 추상화 수준(level of abstraction)이 높다 = 카메라를 높이 올려 개별 디테일 대신 전체 지형을 본다는 뜻입니다.

직관과 반대로, "높다"가 더 자세하다는 의미가 **아닙니다.** 높을수록 디테일은 사라지고 큰 흐름만 남습니다.

| 구분 | 높은 레벨 (high level) | 낮은 레벨 (low level) |
| --- | --- | --- |
| 보는 것 | 숲 (전체 흐름·단계) | 나무 (구체적 코드·도구) |
| 예시 | "ML은 데이터 준비 → 학습 → 배포 순서다" | "중복 제거는 `drop_duplicates()`로, 해시 기준은 이 컬럼" |
| 답하는 질문 | "무슨 단계가 있나?" | "그 단계를 *어떻게* 구현하나?" |

> 한 문장으로: **"높은 레벨에서 ML 수명 주기는 데이터 준비부터 시작한다"** = 세부는 제쳐두고 큰 흐름만 보면 보통 데이터 준비 단계부터 시작한다는 뜻입니다. 뒤따르는 "집계·중복 정리·특성 추출"도 *어떻게* 하는지가 아니라 *무슨 단계가 있는지*의 나열이라 여전히 high-level 설명입니다.

## 2. ML 수명 주기 — 상세 뷰

ML 수명 주기를 한 단계 구체화하면 4개 단계로 흐릅니다. 왼쪽에서 오른쪽으로 진행되고, 전 과정을 하나로 묶어 자동화하는 것이 **Pipeline**입니다.

```
Data Prep → Train & Tune → Deploy & Monitor → Inference
   (준비)      (학습·튜닝)     (배포·모니터링)      (추론/서빙)
```

### ① Data Prep — 데이터 준비

다양한 소스의 데이터를 모아 학습에 쓸 수 있게 다듬는 단계입니다.

- **Data Aggregation**(집계) → **Data Wrangling**(정제·가공) → **Data Balancing**(클래스 불균형 조정)
- 세부: Clean / Normalize / Regularize / Unbias, 그리고 **Detect Bias → Mitigate Bias**(편향 탐지·완화)
- **Data Labeling**(라벨링), **Feature Engineering**(특성 공학 — Transformation·Encoding·Joining)
- 산출물: **Data Artifact** → **Feature Store [Offline]**에 저장

### ② Train & Tune — 학습·튜닝

알고리즘을 골라 모델을 학습시키고 성능을 검증·튜닝하는 단계입니다. **MLflow가 가장 직접적으로 닿는 영역**입니다.

- **Algorithm Selection**(알고리즘 선택) → **Model Training**(학습) → **Metrics Validation**(지표 검증)
- 학습 추적: **Experiments → Trials → Trial Components**(실험/시도/구성요소 단위 기록) ← MLflow의 핵심 기능
- 실행 위치: **Local** / **Hosted**, 이후 **Hyper-param Tuning**(하이퍼파라미터 튜닝)
- 산출물: **Code Artifact**, **Model Artifact** → **Model Registry**(모델 저장소)에 등록

### ③ Deploy & Monitor — 배포·모니터링

검증된 모델을 실제 서비스에 올리고 지속적으로 감시하는 단계입니다.

- **Model Testing**(테스트) → **Model Deployment**(배포)
- 배포 방식: **Batch Transform**(배치 추론) / **Hosted Endpoint**(실시간 엔드포인트)
- **Inference** 결과 → **Feature Store [online]**에 저장
- **Model & Inference Interpretability**(모델·추론 해석 가능성)

### ④ Inference — 추론/서빙

**Users**가 데이터를 보내면 배포된 모델이 예측을 반환합니다. 클라우드 위에서 서빙됩니다.

### 단계를 가로지르는 2가지 핵심 개념

| 개념 | 의미 |
| --- | --- |
| **Artifact Lineage** | Data → Code → Model 산출물의 **계보 추적**. "이 모델이 어떤 데이터·코드로 만들어졌나"를 역추적 |
| **Feature Store (Offline/Online)** | 학습용(offline)·서빙용(online) 특성을 **일관되게 공유** |

> 한 문장으로: ML 수명 주기는 **데이터 준비 → 학습·튜닝 → 배포·모니터링 → 추론**의 4단계이며, MLflow 관점에서 가장 중요한 접점은 **Experiments/Trials(실험 추적)**, **Model Registry**, **Artifact Lineage**입니다.

## 3. MLOps 4대 원칙

MLOps는 한마디로 **"ML을 일반 소프트웨어 개발처럼 신뢰할 수 있게 운영하기 위한 약속"**입니다. 그 토대가 되는 4가지 원칙입니다.

### ① 버전 제어 (Version Control)

코드만이 아니라 **데이터·모델까지 전부 버전을 매겨** 언제든 재현하고 롤백할 수 있게 합니다.

일반 개발은 Git으로 **코드**만 관리하지만, ML 결과는 **코드 + 데이터 + 설정**에 동시에 의존합니다. 같은 코드라도 데이터가 바뀌면 모델 성능이 달라지기 때문입니다. 그래서 "이 모델 = 어떤 코드 + 어떤 데이터 + 어떤 설정"을 모두 기록해야 **재현(reproducibility)·롤백·감사(audit)**가 가능합니다. → 2장의 **Artifact Lineage**가 이를 구현한 것.

> **재현성(reproducibility)** — 같은 입력을 넣으면 항상 같은 결과가 나오는 것. 데이터 처리부터 모델 배포까지 모든 단계에서 보장돼야 합니다.

### ② 자동화 (Automation)

데이터 수집 → 전처리 → 학습 → 검증 → 배포 단계를 **사람 손 없이 자동 실행**되게 만들어 **반복성·일관성·확장성**을 확보합니다.

자동 재학습·재배포를 트리거하는 신호 예시:

- 메시징 / 모니터링·캘린더 이벤트
- **데이터 변경** (새 데이터 유입)
- **모델 학습 코드 변경**, **앱 코드 변경**

> **IaC (Infrastructure as Code, 코드형 인프라)** — 서버·환경 설정을 **코드로 정의**해 어느 환경에서나 동일하게 재현·배포하는 방식. 개발에서 Dockerfile·Terraform을 쓰는 그 발상입니다. 자동화는 IaC와 결합할 때 더 효율적입니다.

### ③ 지속적 X (Continuous X)

자동화 위에서 **끊임없이 돌아가는 4가지 활동**입니다. 일반 개발의 CI/CD를 ML로 확장한 것으로, 시스템 어디든 변경이 생기면 지속적으로 일어납니다.

| 활동 | 일반 개발 | ML에서 확장된 점 |
| --- | --- | --- |
| **CI** 지속적 통합 | 코드 검증·테스트 | + **데이터·모델**까지 검증·테스트 |
| **CD** 지속적 전달 | 코드 배포 | **새로 학습된 모델/예측 서비스** 자동 배포 |
| **CT** 지속적 훈련 | (ML 고유) | 모델을 **자동 재학습**해 재배포 |
| **CM** 지속적 모니터링 | 서비스 상태 감시 | **데이터·모델 성능**을 비즈니스 지표로 감시 |

> **참고** — CT(지속적 훈련)와 CM(지속적 모니터링)은 일반 소프트웨어에는 없는 ML만의 항목입니다. 모델은 시간이 지나면 데이터 분포가 변해 성능이 떨어지기(model drift) 때문입니다.

### ④ 모델 거버넌스 (Model Governance)

ML 시스템 전반을 **관리·통제**하는 규칙과 프로세스로, 기술보다 **사람·정책·책임**에 가까운 원칙입니다.

- **협업**: 데이터 사이언티스트·엔지니어·비즈니스 담당자 간 긴밀한 소통(문서·커뮤니케이션 채널)
- **피드백 루프**: 모델 예측에 대한 피드백을 모아 재학습에 반영
- **보안·규정 준수**: 민감 데이터 보호, 모델·인프라 접근 통제, 컴플라이언스 충족
- **출시 전 검토·승인**: 공정성(fairness)·편향성(bias)·윤리 확인 후 배포

> 한 문장으로: **버전 제어**(재현·롤백) · **자동화**(반복·확장) · **지속적 X**(변화 대응) · **거버넌스**(안전·책임) — 이 4가지가 MLOps의 기둥입니다.

## 4. MLOps의 이점

ML은 본질적으로 실험적·혁신적이라 통제 없이 두면 비용 폭증·실패 위험이 큽니다. MLOps는 그 혼란에 **"지도(map)"를 깔아주는 것**입니다. 아래 3대 이점은 사실 3장의 4대 원칙이 만들어내는 *결과*입니다.

### ① 출시 기간 단축 — "환경 세팅에 쓰던 시간을 없앤다"

- **선언적 구성 파일로 인프라 프로비저닝** = `docker-compose up` / Terraform `apply` 한 줄로 환경이 뜨는 것
- 모델 생성·배포 **자동화** → 출시 시간↓, 운영 비용↓
- 데이터 과학자가 데이터를 빠르게 탐색해 **비즈니스 가치를 더 빨리** 전달

> **선언적(declarative)** = "어떻게 만들지" 절차를 일일이 쓰는 대신 "이런 상태가 되게 해줘"라고 **결과만 선언**하는 방식. 3장의 **IaC**가 바로 이것.

### ② 생산성 향상 — "보일러플레이트와 재사용"

- **개발/실험 환경 표준화** → 새 프로젝트 착수, 여러 프로젝트 전환, 여러 앱에서 **모델 재사용**이 쉬움
- **반복 가능한(repeatable) 실험·학습 프로세스** 구축 (여기서 MLflow가 실험 기록 도구로 등장)
- SW 엔지니어링 팀이 ML 개발 수명 주기 전반에서 **협업·조정** → 효율↑

### ③ 효율적인 모델 배포 — "관측 가능성 + 버전 관리"

- 모델 성능 **모니터링** + 동작 **재현(reproduce)** → 문제 디버깅 가능
- **모델 버전 추적·중앙 관리** → 비즈니스 사례별로 적합한 버전 선택 (= **Model Registry**)
- 모델 워크플로를 **CI/CD 파이프라인에 통합** → 업그레이드·튜닝 후에도 **성능 저하 억제, 품질 유지**

| 이점 | 핵심 효과 | 떠받치는 원칙 |
| --- | --- | --- |
| 출시 기간 단축 | 빠른 착수·배포, 비용↓ | 자동화 + IaC |
| 생산성 향상 | 표준화·재사용·반복 가능 | 자동화 + 버전 제어 |
| 효율적 배포 | 모니터링·재현·버전 관리 | 지속적 X + 버전 제어 |

> **참고** — 백엔드 경험이 있는 개발자라면 이 이점들이 낯설지 않습니다. **IaC·Dev Container·CI/CD·APM·버전 태그**처럼 이미 쓰는 도구들을, ML 특유의 골칫거리(비결정성·데이터 의존·드리프트)에 맞게 적용한 것이기 때문입니다.

## 5. MLOps 구현 — 성숙도 3단계

조직의 자동화 성숙도에 따라 MLOps 구현은 3단계로 나뉩니다. 레벨을 가르는 축은 **"무엇을 자동화했는가"**이고, 결정적 도약은 *"모델을 배포한다 → 모델을 만드는 파이프라인을 배포한다"*로의 사고 전환입니다.

### 레벨 0 — 수동 (Manual)

- 데이터 준비·학습·검증을 **전부 수동**, 단계 간 전환도 손으로 (주피터 노트북 그림)
- 데이터 과학자가 **학습된 모델(아티팩트)을 엔지니어에게 넘기면** 엔지니어가 API에 배포 — **담당 분리**
- 재학습 **연 몇 회**, ML 모델엔 **CI/CD 없음**, **모니터링 없음**

> 개발자 비유: 빌드 산출물을 메신저로 보내면 운영팀이 수동 배포하던 시절. **벽 너머로 던지기(throw over the wall)** 구조.

### 레벨 1 — 학습 파이프라인 자동화

- ⭐ **핵심 전환**: 레벨 0은 *완성된 모델*을 배포했지만, 레벨 1은 **반복 실행되는 학습 파이프라인 자체**를 배포. 그 파이프라인이 알아서 새 모델을 찍어냄
- **새 데이터가 트리거** → 프로덕션에서 **자동 재학습(CT)**
- 개발/프리프로덕션/프로덕션에서 **같은 파이프라인** 사용
- 코드가 **모듈화·재사용·구성 가능**, **중앙 Feature Store** 도입, 실행마다 **메타데이터·재현성 기록**

> 개발자 비유: 손으로 배포하던 걸 **GitHub Actions 워크플로**로 만들어 둔 것. 단 트리거가 "코드 푸시"가 아니라 **"새 데이터 도착"**이고, 결과물이 "배포된 앱"이 아니라 **"방금 재학습된 모델"**.

### 레벨 2 — CI/CD 파이프라인 자동화

- 대상: **분 단위 모델 업데이트, 시간/일 단위 재학습, 수천 서버 동시 재배포**하는 기술 선도 기업
- 레벨 1 전부 + **파이프라인 오케스트레이터** + 여러 모델 추적용 **Model Registry**
- 세 단계가 **규모에 맞게 반복**됨:

| 단계 | 하는 일 | 산출물 | 개발 비유 |
| --- | --- | --- | --- |
| **Build** | 새 알고리즘·모델링 실험, 파이프라인 소스 코드 생성 → repo 저장 | 파이프라인 소스 코드 | 코드 작성 + 커밋 |
| **Deploy** | 소스 빌드·테스트 → 배포할 파이프라인 구성요소 산출 | 배포된 파이프라인 | CI: build & test |
| **Serve** | 예측 서비스로 제공, 라이브 통계 수집 → 다음 실험/재학습 트리거 | 예측 서비스 + 트리거 | CD + 모니터링 피드백 루프 |

> 개발자 비유: 레벨 1이 "내 앱의 CI/CD"라면, 레벨 2는 **"CI/CD 파이프라인 자체를 코드로 만들고 테스트하고 자동 배포하는"** 메타 레벨. 피드백(라이브 통계)이 다음 사이클을 자동으로 깨우는 **닫힌 루프**가 핵심.

### 수동 vs 자동 워크플로

성숙도 레벨은 두 워크플로의 관계로도 그려집니다. **위(수동 탐색)**에서 사람이 손으로 한 바퀴 돌려보고, 검증된 흐름만 **아래(자동)**로 내려보내 굳히는 구조입니다.

- **Manual Exploratory Workflow** (위): Ingest → Prepare/Transform → Process → Store in Feature Store → Detect/Mitigate Bias → Train & Tune → (Associate Lineage·Deposit in Registry) → Deploy → Online Feature Store → Run Inference — 데이터 과학자가 **손으로** 한 단계씩
- **Automated Workflow** (아래): Build Pipeline that integrates Steps → Monitor Models → Retrain Models → Scale Inference — 검증된 단계를 **하나의 파이프라인으로 묶어 자동 루프**

| | 위: Manual Exploratory | 아래: Automated |
| --- | --- | --- |
| 누가 | 데이터 과학자가 **손으로** | 파이프라인이 **자동으로** |
| 목적 | **탐색·실험** | **운영·반복** |
| 성숙도 | 레벨 0 | 레벨 1~2 |

> 한 문장으로: MLOps는 수동 탐색을 없애는 게 아니라, **실험에서 검증된 흐름을 자동화로 승격**시키는 것입니다. "로컬에서 수동 시행착오 → 검증되면 스크립트화 → CI 파이프라인 등록"이라는, 개발자가 늘 하는 그 흐름을 데이터+모델에 적용한 것입니다.

---

## 6. MLOps vs DevOps

한 문장으로 — **MLOps는 "DevOps를 ML에 맞게 확장한 것"**입니다. 뿌리는 같고, ML 특유의 골칫거리를 다루기 위해 몇 가지가 더 붙었습니다.

둘 다 **개발→배포→모니터링을 개선**하고 **팀 간 벽을 허무는 협업 + 자동화**를 추구합니다. 차이는 DevOps가 **코드**를 다루는 반면 MLOps는 **코드 + 데이터 + 모델**을 다룬다는 점입니다.

| 축 | DevOps | MLOps |
| --- | --- | --- |
| 관리 대상 | **코드** | 코드 + **데이터 + 모델** |
| 결과 성격 | 결정적 (테스트 통과 = 끝) | **비결정적** (데이터·시드 따라 달라짐) |
| 품질 저하 | 코드 안 바꾸면 유지 | **가만둬도 나빠짐** (model drift) |
| 허무는 벽 | Dev ↔ Ops | Dev ↔ Ops ↔ **Data Science** |
| 파이프라인 | CI/CD | CI/CD **+ CT(+CM)** |
| "완료"의 정의 | 배포되면 끝 | 배포 후 **정확도 유지·향상**까지 |

> **참고** — DevOps에 없는 MLOps의 가장 큰 추가가 **CT(지속적 훈련)**입니다. 일반 앱은 코드를 안 바꾸면 어제와 똑같이 동작하지만, ML 모델은 **세상의 데이터가 변하면 코드를 안 건드려도 점점 틀려집니다**(model drift). 그래서 배포로 끝이 아니라 **모니터링 → 재학습 → 재배포의 닫힌 루프**가 필수입니다.

---

## 7. 핵심 요약

전체를 한 줄로 꿰면:

> ML 수명 주기는 **데이터 준비 → 학습·튜닝 → 배포·모니터링 → 추론**(2장)으로 흐르고, 이를 **버전 제어·자동화·지속적 X·거버넌스**(3장)라는 4대 원칙으로 운영하면 **출시 단축·생산성·효율적 배포**(4장)의 이점이 나옵니다. 조직은 **수동(레벨0) → 파이프라인 자동화(레벨1) → CI/CD 자동화(레벨2)**(5장) 순으로 성숙해지며, 이 모든 것은 **DevOps를 ML(데이터+모델+드리프트)에 맞게 확장**(6장)한 것입니다.

"높은 레벨"에서 보면 MLOps는 *ML을 일회성 실험이 아니라 반복 가능한 엔지니어링으로 만드는 일*이고, **MLflow**는 그중 ②학습·튜닝 단계의 **실험 추적·모델 레지스트리·아티팩트 관리**를 담당하는 도구입니다.

## 용어집

- **high-level / low-level** — 추상화 수준. 높을수록 큰 그림, 낮을수록 구체적 구현.
- **Artifact Lineage** — 데이터·코드·모델 산출물의 계보 추적.
- **Feature Store** — 학습용(offline)·서빙용(online) 특성 저장소.
- **reproducibility(재현성)** — 같은 입력 → 같은 결과 보장.
- **IaC(Infrastructure as Code)** — 인프라를 코드로 정의·관리.
- **declarative(선언적)** — 절차가 아니라 원하는 결과 상태만 기술.
- **CI / CD / CT / CM** — 지속적 통합 / 전달 / 훈련 / 모니터링.
- **model drift** — 시간이 지나며 데이터 분포가 변해 모델 성능이 저하되는 현상.
- **Model Registry** — 모델 버전을 중앙에서 추적·관리하는 저장소.
- **성숙도 레벨 0/1/2** — 수동 / 학습 파이프라인 자동화 / CI/CD 파이프라인 자동화.
